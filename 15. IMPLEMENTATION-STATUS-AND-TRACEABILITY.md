# 1. Purpose and authority

Documents 01–14 remain the authoritative product and engineering baseline. This
register reports implementation reality. It does not modify requirement meaning,
scope, priority, acceptance criteria, approval, or authority. In a conflict, the
source documents govern and the deviation remains open until approved change
control resolves it.

Code existence alone does not make a requirement fully implemented. A requirement
may be marked implemented only when applicable code, tests, failure paths,
documentation, and acceptance evidence exist. Local demonstrations and manual
checks are named as such and are not production evidence.

# 2. Status-classification model

| Classification | Meaning |
| --- | --- |
| Implemented and verified | Applicable code, automated tests, failure paths, documentation, and acceptance evidence exist for the stated narrow capability. |
| Partially implemented | Some target behavior exists, but required scope, controls, persistence, failure handling, or acceptance evidence is absent. |
| Verification-only prototype | Deterministic or local-only behavior exists to prove wiring or testability, not product behavior. |
| Temporary compatibility behavior | A short-lived contract or route supports incremental development and is expected to be replaced. |
| Not started | No material implementation evidence exists. |
| Deferred | The baseline or an approved implementation decision explicitly places the capability later. |

# 4. Current implementation-stage summary

The application is a Phase 1 TypeScript monorepo foundation with a synchronous,
provider-backed chat slice. It proves browser → Vite proxy → Fastify → selected
provider wiring, canonical public problems, request correlation, deterministic tests,
and a development-
installable Android PWA shell. It is not the documented Gexor MVP, is not production
ready, and provides database-backed authenticated personal workspaces and durable canonical acceptance, but no streaming,
streaming, context, memory, usage, audit, or operational platform.

The following narrow capabilities are implemented and verified at the application
baseline:

- npm workspace monorepo and strict TypeScript foundation;
- API, web, and contracts workspaces;
- shared public problem, temporary chat, message-submission, and runtime-execution
  contracts;
- Fastify application, dependency injection, compatibility health, and versioned
  health;
- strict request validation, application/problem+json responses, safe field errors,

  failed, timed_out, and cancelled states;
- timestamped correlated snapshots with safe provider/model, response, and failure
  outcomes;
- shared runtime executor used by canonical submission and temporary POST /chat;
- canonical POST /api/v1/conversations/{conversationId}/messages returning 202
  and progressing asynchronously;
- canonical GET /api/v1/executions/{executionId} endpoint;
- React browser chat, recoverable errors, abort timeout, late-result guard, and

  timing-safe verification;

“Implemented and verified” here does not assert satisfaction of the canonical MVP

# 6. Partially implemented capabilities

## 6.1 Provider connections

Current: authenticated workspace owners can list a persisted provider/model catalogue and create, validate, select, revoke, and rotate workspace-owned connections. Only opaque protected credential references are stored; public snapshots exclude references. Canonical execution requires and resolves the workspace selection. Lifecycle audit rows record actor, workspace, connection, action, and time.

Missing: production secret-manager integration, distributed reference-access audit export, provider health monitoring, and broader model catalogue operations. The local-env:configured reference and temporary /chat route remain explicit local-development compatibility behavior.

## 6.2 Runtime execution

Current: canonical and compatibility requests share an in-memory executor. Accepted
executions progress asynchronously through preparation and provider dispatch to
completed, failed, or timed_out snapshots; cancellation transitions are represented
and guarded. The narrow `accepted`, `dispatching`, and `timed_out` vocabulary maps
to the documented baseline's creation, provider-pending, and terminal-failure concepts.
Snapshots preserve correlation, timestamps, safe provider/model outcome,
completed response, or controlled failure.

Missing: durable runtime state and the broader documented pipeline stages, effective-
configuration/context/prompt snapshots, route-decision and usage records,
background-job handoff, reconciliation, idempotent finalization, and Snapshot Lock.

## 6.3 Chat API

Current: temporary synchronous POST /chat returns 200 through the shared runtime
executor. Canonical versioned submission returns 202 with transient identities and
starts asynchronous in-process execution; execution reads expose current snapshots.

Missing: durable conversation/message/execution
resources, stream URL, durable idempotency key,
SSE, cancellation, retry, and regeneration.

## 6.4 Error contract

Current: shared provider-neutral application/problem+json with stable type, title,
status, code, safe detail, requestId, retryable, optional safe field errors, and
matching
X-Request-Id response correlation.

Missing: published resolvable error documentation, full canonical category coverage,
and distributed trace correlation.

## 6.5 Mobile UX

Current: responsive authenticated workspace with mobile navigation, conversation history, execution status, provider settings, recoverable errors, and standalone development PWA installation.

Missing: projects, memory, usage,
memory, usage, streaming/cancellation state, and complete WCAG 2.2 AA verification.

## 6.6 Authentication and personal workspace

Current: versioned register, login, logout, and current-user endpoints; strict bearer-session preHandler middleware; generic login failures; idempotent logout; server-evaluated expiry and revocation; atomic-in-process identity/workspace initialization with rollback; one active personal workspace and owner membership per account; and fail-closed explicit workspace authorization on canonical message submission and execution reads. Deterministic tests cover success, validation, enumeration resistance, expiry, revocation, rollback, idempotent provisioning, and cross-workspace denial.

Current additionally includes durable SQLite identity, session revocation, workspace, conversation, message, execution, provider connection, idempotency, and outbox state plus browser registration, login, logout, and refresh restoration. Missing: email verification, password recovery, MFA/passkeys, production throttling, durable audit, and production assurance. The temporary compatibility routes remain unauthenticated by design and outside the canonical product boundary.

# 11. Security and secret-handling status

Local controls now also include a narrow password and session foundation: versioned Node scrypt with unique salts, timing-safe verification, opaque random session tokens, and one-way token hashing. Public contracts exclude credential and token hashes. These controls are exposed through a process-local authenticated API boundary but are not persistent or production-ready.

Local controls include ignored environment files, a placeholder-only .env.example,
startup validation, API-key trimming, provider headers kept server-side, normalized
public errors, and tests that avoid credential response leakage. These are local-
development controls only. They do not satisfy the secret-manager, workspace-
isolation, authorization, encryption, rotation, access-audit, security scanning, or
production assurance requirements. Credentials must never enter source control,
logs, client bundles, examples, queues, or public errors.

# 12. Testing and verification evidence

Automated Version 1 evidence comprises 109 API tests and 4 web tests,
strict workspace typechecks, web/contracts builds where applicable, and root
<code>npm run verify</code>. Tests cover schemas, normalized errors, provider adapters,
factory/configuration, dependency injection, temporary routes, React behavior,
timeout/abort, duplicate submit, and a React-to-real-Fastify bridge.

Manual evidence includes the authenticated local end-to-end flow and prior Gemini browser integration and Android standalone
installation. It does not establish live-provider CI, end-to-end authentication,
tenant isolation, database, performance, accessibility, security, resilience,
backup/restore, deployment, or operational acceptance.
