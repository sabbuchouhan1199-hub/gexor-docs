# 1. Purpose and authority

Documents 01–14 remain the authoritative product and engineering baseline. This
register reports implementation reality. It does not modify requirement meaning,
scope, priority, acceptance criteria, approval, or authority. In a conflict, the
source documents govern and the deviation remains open until approved change
control resolves it.

Code existence alone does not make a requirement fully implemented. A requirement
may be marked implemented only when applicable code, tests, failure paths,
documentation, and acceptance evidence exist. Local demonstrations and manual
checks are named as such and are not production evidence.

# 2. Status-classification model

| Classification | Meaning |
| --- | --- |
| Implemented and verified | Applicable code, automated tests, failure paths, documentation, and acceptance evidence exist for the stated narrow capability. |
| Partially implemented | Some target behavior exists, but required scope, controls, persistence, failure handling, or acceptance evidence is absent. |
| Verification-only prototype | Deterministic or local-only behavior exists to prove wiring or testability, not product behavior. |
| Temporary compatibility behavior | A short-lived contract or route supports incremental development and is expected to be replaced. |
| Not started | No material implementation evidence exists. |
| Deferred | The baseline or an approved implementation decision explicitly places the capability later. |
| Blocked by prerequisite | Work cannot safely satisfy its requirement until a named dependency exists. |
| Superseded | An approved replacement makes the earlier implementation or plan non-current. |

Status applies only to the capability wording in this register. It is not a release
or compliance declaration.

# 4. Current implementation-stage summary

The application is a Phase 1 TypeScript monorepo foundation with a synchronous,
provider-backed chat slice. It proves browser → Vite proxy → Fastify → selected
provider wiring, normalized local failures, deterministic tests, and a development-
installable Android PWA shell. It is not the documented Gexor MVP, is not production
ready, and provides no authenticated workspace, persistence, runtime state machine,
streaming, context, memory, usage, audit, or operational platform.

# 5. Implemented capabilities

The following narrow capabilities are implemented and verified at the application
baseline:

- npm workspace monorepo and strict TypeScript foundation;
- API, web, and contracts workspaces;
- minimal shared chat request/response types;
- Fastify application, dependency injection, and health endpoint;
- strict chat request validation and safe normalized local/provider errors;
- provider-independent TextProvider interface;
- Ollama and Gemini non-streaming adapters with timeouts and response validation;
- configuration-selected provider factory;
- temporary provider-backed POST /chat route;
- React browser chat, recoverable errors, abort timeout, late-result guard, and
  duplicate-submit prevention;
- browser-to-Fastify integration test and deterministic provider/API tests;
- Vite development proxy;
- manifest, original icons, standalone display metadata, and manually verified
  Android development installation;
- workspace typechecks, tests, builds, and root verify workflow.

“Implemented and verified” here does not assert satisfaction of the canonical MVP
resource, authorization, persistence, streaming, or operational requirements.

# 6. Partially implemented capabilities

## 6.1 Provider connections

Current: process-level environment configuration, local credential loading, Gemini
and Ollama adapters, model names, timeouts, and factory selection.

Missing: workspace-owned provider connections, protected credential reference,
secret-manager boundary, connect/test/revoke/rotate lifecycle, per-workspace model
selection, health state, and provider-connection audit. This is partially
implemented, not a BYOK provider-connection implementation.

## 6.2 Runtime execution

Current: validated input, direct adapter dispatch, provider timeout, and normalized
success/failure.

Missing: runtime execution identity and durable state machine, effective-
configuration/context/prompt snapshots, route-decision and usage records,
background-job handoff, reconciliation, idempotent finalization, and Snapshot Lock.

## 6.3 Chat API

Current: temporary synchronous POST /chat returning 200 with a reply.

Missing: /api/v1, authentication and workspace scope, conversation/message
resources, 202 Accepted, execution resource, stream URL, request ID, idempotency key,
SSE, cancellation, retry, and regeneration.

## 6.4 Error contract

Current: safe normalized JSON with code, message, and status; internal ProviderError
also records retryability.

Missing: application/problem+json, public requestId and retryable fields, canonical
error type URI, safe field-error detail, and correlation support.

## 6.5 Mobile UX

Current: responsive single-chat view and standalone development PWA installation.

Missing: workspace shell, projects/conversation navigation, provider/model controls,
memory, usage, streaming/cancellation state, and complete WCAG 2.2 AA verification.

# 8. Not-started MVP capabilities

No material implementation evidence exists for:

- authentication/session lifecycle, personal workspace provisioning, membership,
  roles, permissions, tenant isolation, projects, conversations, messages, database,
  migrations, and persistent repositories;
- idempotent message acceptance, outbox/inbox, runtime execution state machine,
  SSE/reconnect/replay, cancellation, route-decision persistence, prompt enhancement,
  intent classification, context retrieval, token budgeting, snapshots, structured
  memory, memory candidates, and Snapshot Lock;
- uploads, parsing/chunking, keyword/vector/hybrid retrieval, usage/token/cost
  records, reconciliation, quotas, spending controls, durable jobs, workers, queue,
  audit events, notifications, export, retention/deletion, backup, and recovery;
- distributed metrics/logs/traces, production deployment, CI/CD security gates,
  and production incident-response readiness.

# 9. Explicitly deferred capabilities

Documents 01–14 defer or place beyond the locked MVP capabilities including
autonomous agents, general-purpose tool orchestration, third-party integration
marketplace, organization collaboration beyond personal-workspace compatibility,
unsupported file formats and OCR unless approved, provider-managed billing resale,
proprietary foundation-model training, native mobile applications, and full offline
PWA behavior. The present service-worker/offline step is also intentionally deferred.

# 11. Security and secret-handling status

Local controls include ignored environment files, a placeholder-only .env.example,
startup validation, API-key trimming, provider headers kept server-side, normalized
public errors, and tests that avoid credential response leakage. These are local-
development controls only. They do not satisfy the secret-manager, workspace-
isolation, authorization, encryption, rotation, access-audit, security scanning, or
production assurance requirements. Credentials must never enter source control,
logs, client bundles, examples, queues, or public errors.

# 12. Testing and verification evidence

Automated evidence at <code>3761bb6</code> comprises 55 API tests and 10 web tests,
strict workspace typechecks, web/contracts builds where applicable, and root
<code>npm run verify</code>. Tests cover schemas, normalized errors, provider adapters,
factory/configuration, dependency injection, temporary routes, React behavior,
timeout/abort, duplicate submit, and a React-to-real-Fastify bridge.

Manual evidence records successful Gemini browser integration and Android standalone
installation. It does not establish live-provider CI, end-to-end authentication,
tenant isolation, database, performance, accessibility, security, resilience,
backup/restore, deployment, or operational acceptance.
